# ez-toolkit-bin

Release artifact repository for Elvin Zeng's **ez-toolkit** CLI toolkit
(`ezt`, `ezlog`, `ezcrypt`, and friends). This repository hosts the published
binaries, their signatures, and the metadata needed by `install.sh` and `ezt`
to discover, download, and verify each release.

Source code lives in the private `ezcli` repository. Release artifacts are
produced by `ezcli`'s release pipeline (`scripts/release`) and pushed here.

## Repository layout

```
ez-toolkit-bin/
├── README.md                     — this file
├── install.sh                    — POSIX sh bootstrap installer
├── ezcrypt_public.pem            — trust root (ECC + ML-DSA hybrid, level set at keygen)
└── meta/
    ├── release-manifest.toml     — catalog of the current release
    ├── release-manifest.toml.ezg — ezcrypt signature of the manifest
    ├── bootstrap-index.sh        — shell-friendly per-platform package index
    └── bootstrap-index.sh.ezg    — ezcrypt signature of the index
```

All files under `meta/` are **generated by the release pipeline** in the
`ezcli` repository (`scripts/release publish`). Do not edit them by hand.
Binaries themselves are distributed as GitHub Release assets; this repo only
hosts the metadata and signatures needed to bootstrap `ezt`.

## Installation

```sh
export EZTOOLKIT_ROOT="$HOME/.eztoolkit"
curl -fsSL https://raw.githubusercontent.com/elvinzeng/ez-toolkit-bin/master/install.sh | sh
export PATH="$EZTOOLKIT_ROOT/bin:$PATH"
```

Add the two `export` lines to your shell profile (`~/.bashrc`, `~/.zshrc`,
etc.) so the binaries stay on your `PATH` between sessions. After bootstrap,
use `ezt install <cmd>`, `ezt upgrade`, and `ezt ls-remote` for day-to-day
management.

## Trust model (two-phase)

`install.sh` bootstraps from nothing, so it cannot rely on ezcrypt signatures
up-front. It uses a two-phase trust model instead (see the
`ezcli/docs/superpowers/specs/2026-04-11-release-pipeline-design.md` spec,
section 13, for the full rationale):

| Phase | Trust source                              | What it catches                                        |
|-------|-------------------------------------------|--------------------------------------------------------|
| 1     | HTTPS + SHA256 from `bootstrap-index.sh`  | In-flight corruption, opportunistic tampering          |
| 2     | `ezcrypt verify` against `ezcrypt_public.pem` | Full repository compromise (provided the key holds up) |

Phase 1 fetches `meta/bootstrap-index.sh` over HTTPS, sources it with `.`
(dot), reads the platform-specific package filename and SHA256, downloads the
package from the latest GitHub Release, and checks the SHA256. Phase 2 uses
the newly-installed `ezcrypt` to verify the release manifest signature and
every installed binary's `.ezg` sibling against `ezcrypt_public.pem`.

**`ezcrypt_public.pem` is the trust root.** It is committed to this
repository, downloaded during Phase 1, and compared against the copy
hardcoded inside `ezt` on every subsequent run. Any mismatch causes `ezt` to
refuse to proceed.

If Phase 2 ever warns about a signature failure after a fresh install, treat
the installation as untrusted. Investigate before running the installed
binaries — see `ezt verify` for a per-file breakdown.

## Key rotation

If the signing key is ever rotated, a new `ezcrypt_public.pem` will be
committed here alongside a release that is signed by the new key. The
hardcoded key inside `ezt` must be updated in the same release cycle so the
two copies stay in sync; otherwise `ezt` will (correctly) reject the new
artifacts as untrusted.
