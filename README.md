# ez-toolkit-bin

Release artifacts for **ez-toolkit**, Elvin Zeng's CLI suite
(`ezt`, `ezlog`, `ezcrypt`, `ezsecret`, `ezsecretd`). This repository hosts the
published metadata and signatures needed by `install.sh` and `ezt` to discover,
download, and verify each release.

## Repository layout

```
ez-toolkit-bin/
├── README.md                     — this file
├── install.sh                    — POSIX sh bootstrap installer
├── ezcrypt_public.pem            — trust root (ECC + ML-DSA hybrid)
└── meta/
    ├── release-manifest.toml     — catalog of the current release
    ├── release-manifest.toml.ezg — signature of the manifest
    ├── bootstrap-index.conf      — shell-friendly per-platform package index
    └── bootstrap-index.conf.ezg  — signature of the index
```

All files under `meta/` are generated by the release pipeline. Do not edit
them by hand. Binaries themselves are distributed as GitHub Release assets;
this repo only hosts the metadata and signatures needed to bootstrap `ezt`.

## Installation

First, add the following to your shell profile (`~/.bashrc`, `~/.zshrc`,
etc.):

```sh
export EZTOOLKIT_ROOT="$HOME/.eztoolkit"
export PATH="$EZTOOLKIT_ROOT/bin:$PATH"
```

Then reload your shell (or `source` the profile) and run the bootstrap
installer:

```sh
curl -fsSL https://raw.githubusercontent.com/elvinzeng/ez-toolkit-bin/master/install.sh | sh
```

After bootstrap,
use `ezt install <cmd>`, `ezt upgrade`, and `ezt ls-remote` for day-to-day
management.

## Trust model (two-phase)

`install.sh` bootstraps from nothing, so it cannot rely on cryptographic
signatures up-front. It uses a two-phase trust model:

| Phase | Trust source                              | What it catches                                        |
|-------|-------------------------------------------|--------------------------------------------------------|
| 1     | HTTPS + SHA256 from `bootstrap-index.conf`  | In-flight corruption, opportunistic tampering          |
| 2     | `ezt` built-in signature verification       | Full repository compromise (provided the key holds up) |

Phase 1 fetches `meta/bootstrap-index.conf` over HTTPS, extracts the
platform-specific package filename and SHA256 as pure data (never executed
as shell code), downloads the package from the latest GitHub Release, and
checks the SHA256. Phase 2 is performed by `ezt` itself, which has the
signing public key embedded at build time and verifies the release manifest
signature and every installed binary's `.ezg` sibling directly — no external
tool is needed.

**`ezcrypt_public.pem` is the trust root.** It is committed to this
repository. During Phase 1 the installer downloads it into a staging
directory for one-shot use — `ezcrypt verify -k <path>` needs a
filesystem path to validate the staged manifest and binaries. The
file is removed alongside the rest of the staging directory once
install completes (success or failure), so nothing persists in
`$EZTOOLKIT_ROOT`. On every subsequent `ezt` run, the public key
embedded in the `ezt` binary itself is the trust root: `ezt` always
re-fetches the published `ezcrypt_public.pem` from this repository
and compares it against its embedded copy. Any mismatch causes `ezt`
to refuse to proceed.

If Phase 2 ever warns about a signature failure after a fresh install, treat
the installation as untrusted. Investigate before running the installed
binaries — see `ezt verify-install` for a per-file breakdown.

## Key rotation

If the signing key is ever rotated, a new `ezcrypt_public.pem` will be
committed here alongside a release that is signed by the new key. The
hardcoded key inside `ezt` must be updated in the same release cycle so the
two copies stay in sync; otherwise `ezt` will (correctly) reject the new
artifacts as untrusted.
